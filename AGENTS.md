# AirGap QR Transfer Engineering Guide

本文件是本项目后续所有编码任务的长期工程约束。任何实现、重构、测试和文档修改都应优先遵守这里的规则。

## 项目定位

- 项目名称：AirGap QR Transfer
- 中文名称：基于动态二维码的跨平台离线文件摆渡系统
- 目标平台：Windows 与 Android
- 核心目标：在无网络、无 U 盘、无光盘等传输介质的物理隔离环境下，通过一端屏幕连续显示动态二维码、另一端摄像头连续扫描，实现文件离线传输。

## 核心场景

第一版最终应支持以下使用场景，但 MVP 以单向传输链路为优先：

- Windows 发送文件，Android 接收文件。
- Android 发送文件，Windows 接收文件。
- Windows 与 Windows 互传。
- Android 与 Android 互传。

## 技术路线

- 使用 Qt 6、QML、C++20。
- UI 层使用 QML。
- 核心传输协议必须是独立 C++ 模块，不依赖 QML 或具体 UI。
- 核心协议应尽量不依赖 Qt UI 模块，优先保持可单元测试、可复用。
- 摄像头采集优先使用 Qt Multimedia。
- 二维码生成和识别必须封装为独立 adapter，避免业务逻辑绑定具体二维码库，便于后续替换底层实现。
- 第一版只实现单向传输：发送端循环播放二维码帧，接收端扫描并重组。
- 第一版不实现网络传输、蓝牙传输、Wi-Fi 传输、U 盘导入导出。

## 安全约束

- 应用不得主动访问网络。
- 不要加入联网更新、远程日志、云同步、遥测统计、崩溃上传等联网能力。
- 不要把文件内容写入系统临时目录之外的不可控位置。
- 接收文件保存前必须进行 SHA-256 完整性校验。
- 每个传输会话必须有 `session_id`，解析和重组时必须用它隔离不同文件或不同会话的帧。
- 所有解析函数必须处理错误输入，返回明确错误，不允许因畸形帧、缺失字段、越界索引或损坏数据而崩溃。
- 文件名、路径、manifest 字段等外部输入都必须视为不可信输入。

## 第一版 MVP 功能

MVP 按小阶段推进，不要一次性实现过多功能。目标能力包括：

- 选择单个文件。
- 生成文件 manifest。
- 将文件切分为固定大小 chunk。
- 每个 chunk 编码为一个二维码帧。
- 发送端全屏循环播放二维码帧。
- 接收端通过摄像头识别二维码帧。
- 接收端根据 `session_id`、`chunk_index` 去重和收集。
- 收齐所有 chunk 后重组文件。
- 校验 SHA-256。
- 显示成功或失败结果。

## 推荐工程结构

```text
/src/core   传输协议、分片、manifest、校验、重组逻辑
/src/qr     二维码编码/解码适配层
/src/app    Qt/QML 应用入口和 UI 控制器
/tests      核心协议单元测试
/docs       协议说明、开发说明、测试说明
```

如需调整结构，必须保持核心协议、二维码适配层、应用 UI 层三者边界清晰。

## 协议与数据约定

- 所有二进制帧都必须包含版本号。
- 帧格式应在 `docs/protocol.md` 中说明，包括字段顺序、整数端序、长度限制、校验策略和错误处理规则。
- manifest 至少应能表达文件名、文件大小、chunk 大小、chunk 总数、SHA-256、`session_id` 等重组所需信息。
- chunk 帧必须包含 `session_id`、`chunk_index`、`chunk_count` 或可验证总数、payload 长度和 payload 数据。
- 接收端必须对 `chunk_index` 范围、payload 长度、总大小、重复帧、跨会话帧和版本不兼容进行显式处理。
- 重组完成后，必须对完整文件计算 SHA-256，并与 manifest 中的哈希一致后才视为成功。

## 代码约定

- C++ 标准使用 C++20。
- 核心协议不得直接依赖 QML。
- 核心协议 API 应便于单元测试，不应要求真实摄像头、真实屏幕或 QML 环境。
- QR 编码/解码通过 adapter 接口调用，不在核心协议中直接调用具体二维码库。
- 优先使用清晰的小模块和显式错误类型，避免隐藏全局状态。
- 所有涉及二进制解析、长度计算、索引计算、文件大小计算的代码都要防御溢出和越界。
- 文件 IO 与协议编解码尽量分层，便于单独测试。
- 修改已有代码时遵循项目现有风格；没有既有风格时，采用简洁、可测试、保守的实现。

## 测试要求

- 核心协议必须有单元测试。
- 分片、manifest、帧编码、帧解析、去重、重组、SHA-256 校验都应逐步补齐测试。
- 错误输入测试与正常路径测试同等重要，至少覆盖损坏帧、版本不兼容、长度不匹配、索引越界、重复 chunk、缺失 chunk、哈希不匹配。
- UI、摄像头和二维码库相关能力可以通过 adapter mock 或小型集成测试逐步覆盖。

## 文档要求

- `README.md` 必须说明如何构建 Windows 和 Android 版本。
- `docs/protocol.md` 必须说明二维码帧协议。
- `docs` 中的开发说明和测试说明应随着实现阶段同步更新。
- 每次修改后，最终回复必须说明构建和测试结果；如果未能运行，也要明确说明原因。

## 工作节奏

- 每次任务只完成一个明确阶段，保持变更范围小而可验证。
- 优先先落核心协议和测试，再接入二维码 adapter，最后接入 QML UI 与摄像头流程。
- 不要提前加入 MVP 之外的传输方式或联网能力。
- 遇到需求冲突时，优先保证安全约束、协议独立性和可测试性。

## 完成标准

一个阶段性任务完成时，至少应满足：

- 项目能够构建，或清楚说明当前不能构建的缺口。
- 与本阶段相关的核心协议逻辑有单元测试。
- README 和协议文档在涉及构建方式或协议格式变化时同步更新。
- 最终交付说明包含本次变更内容、构建结果和测试结果。
