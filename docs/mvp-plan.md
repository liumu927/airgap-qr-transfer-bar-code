# AirGap QR Transfer MVP 开发计划

## 总体原则

MVP 采用小阶段推进。每个阶段都应有明确产物、可验证结果和测试说明，避免一次性实现过多功能。

优先级顺序：

1. 核心协议库和单元测试。
2. 二维码编码/解码 adapter。
3. 发送端 UI。
4. 接收端 UI。
5. 端到端测试。
6. Android 打包和权限处理。

MVP 不做网络、蓝牙、Wi-Fi、U 盘导入导出、云同步、自动更新、遥测和远程日志。

## 阶段 1：核心协议库

目标：建立不依赖 QML、不依赖摄像头、不依赖二维码库的 C++20 核心协议模块。

建议范围：

- 创建 `/src/core`。
- 定义 manifest、data、end 帧数据结构。
- 实现通用帧头编码和解析。
- 实现 CRC32。
- 接入或封装 SHA-256。
- 实现 `session_id` 与 `file_id` 生成策略。
- 实现固定大小 chunk 计算逻辑。
- 实现接收端 chunk 去重和收集状态。
- 实现重组后的 SHA-256 校验。
- 创建 `/tests/core`。

验收标准：

- 可构建核心库。
- 有核心单元测试。
- 测试覆盖正常编码/解析、CRC32 错误、版本不兼容、长度错误、chunk 越界、重复 chunk、SHA-256 不匹配。
- 不依赖 QML、Qt Multimedia 或真实文件选择 UI。

建议先做的最小任务：

- 建立 CMake 项目骨架。
- 实现 frame codec 和协议单元测试。
- 暂不接入真实二维码。

## 阶段 2：二维码编码/解码适配

目标：通过 adapter 把核心协议二进制帧转换为二维码图像，并从图像中识别回二进制帧。

建议范围：

- 创建 `/src/qr`。
- 定义 `QrEncoder` 和 `QrDecoder` 抽象接口。
- 选择第一版二维码库，并封装在 adapter 内部。
- 支持二进制 payload 到二维码图像的编码。
- 支持摄像头图像或静态图像到 payload 的解码。
- 增加 QR 容量限制探测或配置。
- 为测试提供 mock adapter 或静态图片测试。

验收标准：

- 核心协议不直接包含二维码库头文件。
- adapter 能编码和解码一组已知 payload。
- 明确第一版建议 chunk 大小。
- 当 payload 超出二维码容量时能返回错误，而不是崩溃或静默截断。

风险点：

- 不同平台二维码库可用性不同。
- 二维码容量、纠错等级、屏幕尺寸和摄像头质量会共同影响实际吞吐。
- 需要保留替换底层库的空间。

## 阶段 3：发送端 UI

目标：实现用户可操作的发送流程：选择文件、生成帧、全屏循环播放二维码。

建议范围：

- 创建 `/src/app` 和 QML 页面结构。
- 实现文件选择。
- 调用核心协议生成 manifest/data/end 帧。
- 调用 QR adapter 生成二维码图像。
- 实现发送进度与帧序号显示。
- 实现全屏播放模式。
- 实现开始、暂停、停止。
- 防止屏幕休眠的策略可作为后续增强，第一版先记录平台差异。

验收标准：

- Windows 桌面端能选择单个文件并循环播放二维码帧。
- UI 不包含任何联网能力。
- 文件不可读、文件过大、二维码容量不足时有明确错误提示。
- 播放序列包含 manifest、data、end。

建议限制：

- 第一版可以先以 Windows 发送端为主要调试入口。
- Android 发送端 UI 可在基础发送流程稳定后适配。

## 阶段 4：接收端 UI

目标：实现摄像头扫描、接收进度、重组校验和结果展示。

建议范围：

- 使用 Qt Multimedia 打开摄像头。
- 将摄像头帧传入 QR decoder adapter。
- 将解码 payload 交给核心协议接收会话。
- 显示当前 session、文件名、文件大小、已收 chunk 数和百分比。
- 收齐后重组文件并校验 SHA-256。
- 校验成功后允许用户选择保存位置。
- 校验失败或会话冲突时显示明确错误。

验收标准：

- 接收端能识别循环播放的二维码帧。
- 能按 `session_id`、`file_id`、`chunk_index` 去重。
- 能处理重复帧和漏帧，等待后续循环补齐。
- SHA-256 校验通过后才显示成功。
- Android 上相机权限被拒绝时有明确状态。

建议限制：

- 第一版仅支持一个活动接收任务。
- 暂不做后台接收。
- 暂不支持多摄像头复杂选择，Windows 可先选择默认摄像头。

## 阶段 5：端到端测试

目标：验证 Windows 与 Android 之间的真实传输闭环，并沉淀可重复测试方法。

建议范围：

- 使用小文件进行端到端测试。
- 覆盖文本文件、二进制文件和不同文件名。
- 测试接收端中途开始扫描。
- 测试手动遮挡导致漏帧后是否能在下一轮补齐。
- 测试重复帧、损坏帧和跨 session 帧。
- 记录不同 chunk 大小、二维码纠错等级和播放帧率下的成功率。
- 编写 `/docs/testing.md` 或在 README 中补充测试步骤。

验收标准：

- 至少完成 Windows 发送到 Android 接收的一条真实链路。
- 至少完成一个同平台链路测试，例如 Windows 到 Windows。
- 每次端到端测试记录文件大小、chunk 大小、帧率、耗时和结果。
- 失败用例有可复现描述。

## 阶段 6：Android 打包和权限处理

目标：完成 Android 端可安装、可运行、权限行为清晰的 MVP 包。

建议范围：

- 配置 Qt Android 构建。
- 配置 Android manifest 中的相机权限。
- 处理运行时相机权限申请和拒绝。
- 检查文件选择与保存路径策略。
- 检查屏幕方向、全屏显示和屏幕常亮需求。
- 验证应用不声明网络权限。
- 生成调试 APK。

验收标准：

- Android APK 可安装并打开。
- 接收模式能请求相机权限。
- 权限拒绝不会崩溃。
- Android 发送模式能全屏显示二维码。
- 应用 manifest 不包含不必要的网络权限。
- README 记录 Windows 和 Android 构建步骤。

## 推荐下一步任务

当前“阶段 1：核心协议库”和“阶段 2：二维码编码/解码适配”的基础切片已完成；“阶段 3：发送端 UI”已建立 Qt/QML 可选目标和发送端骨架，并已使用官方 Qt 6.8.3 MSVC 2022 x64 Kit 完成 Release 构建和部署。发送帧构建流程已抽为可测试的 `airgap_app_core`，并覆盖 manifest/data/end 序列、非法文件名、空文件和 QR 容量不足。

“阶段 4：接收端 UI”的前置 app-core 切片已完成：`ReceiveFrameCollector` 可以接收 QR decoder 输出的协议 payload，完成帧解析、CRC32 校验、manifest/data/end 分发、session/file 过滤、chunk 去重、end 帧确认、重组和 SHA-256 校验。该切片已覆盖乱序接收、重复 data 幂等、坏 payload、跨 session data 和 end 早于 manifest 等场景。

“阶段 4：接收端 UI”的摄像头识别管线切片已完成：Qt app 现在依赖 `Qt6::Multimedia`，QML 中增加发送/接收模式切换，接收页使用 `Camera`、`CaptureSession` 和 `VideoOutput` 打开默认摄像头预览，并通过 `ReceiveController` 管理扫描状态、摄像头可用性、chunk 进度和错误提示。QML `VideoOutput` 会把 `videoSink` 交给 C++，`ReceiveController` 节流读取 `QVideoFrame`、转灰度图、调用 ZXing adapter，并把识别出的 payload 喂给 `ReceiveFrameCollector`。接收完成后，保存按钮只会在 SHA-256 校验通过后启用，并把内存中的重组结果写入用户选择的本地路径。当前还没有完成真实摄像头端到端验收。

下一步建议继续“阶段 4：接收端 UI”的 Qt Multimedia 接入切片：

- 做 Windows 桌面端扫描发送端窗口的本机闭环，记录小文本/小二进制文件的帧数、耗时和成功率。
- 根据真实识别效果调整 QR 显示尺寸、解码节流间隔、ZXing reader 参数和 chunk 大小，再进入 Android 权限和打包。

这个任务能把已经稳定的协议层、二维码编码/解码层和接收 app-core 接到可操作界面上，同时仍然保持范围可控，暂不碰 Android 打包。

## 阶段 5B：手动反馈重传优化

目标：解决大文件或高帧数传输尾部等待少量漏帧的问题。

建议范围：
- 在核心层支持 missing_request 反馈帧编码和解析。
- 接收端生成缺失 chunk range。
- 接收端显示反馈二维码。
- 发送端扫描反馈二维码并验证 `session_id` / `file_id`。
- 发送端构建临时重传播放列表，只播放 `manifest + 缺失 data chunk + end`。
- 保留回到全量循环播放的入口。

验收标准：
- 核心反馈帧有单元测试。
- 接收端能在未完成时生成缺失清单。
- 发送端拒绝不匹配当前任务的反馈二维码。
- 500+ 帧文件在人工漏帧后，可通过反馈重传明显缩短补齐时间。
- 不新增网络、蓝牙、Wi-Fi 或后台通信。

详细设计见 `docs/manual-feedback-resend.md`。
